# UI/UX Log Sheet

Timestamp: 2026-03-22 13:00:00

## Summary

Diagnosed the `Create Project` failure on `/projects/new` and traced it to local PostgreSQL configuration rather than the upload flow itself. Updated error handling so database connectivity failures surface as a specific database-availability error instead of a generic unexpected-error message.

## Error Observed

- Clicking `Create Project` after uploading valid PNG screenshots failed on the new project page
- The UI showed only `An unexpected error occurred.`
- The flow did not proceed into the editor

## Root Cause

- The create flow attempts guest-session creation when no authenticated session is present
- `POST /api/v1/guest-sessions` failed before project creation because Prisma could not initialize against the configured database
- The server log showed `User was denied access on the database '(not available)'`
- This came from local PostgreSQL setup:
  - Postgres was running
  - the local role `postgres` did not exist
  - the configured database `parallax_story_composer` did not exist yet
- Because the thrown error was a raw `PrismaClientInitializationError`, the shared error mapper returned a generic `INTERNAL_SERVER_ERROR`

## Fix Applied

- Infrastructure fix:
  - created local PostgreSQL role `postgres` with password `postgres`
  - created database `parallax_story_composer`
  - applied the pending Prisma migration so the schema is ready
- Application fix:
  - updated `src/core/errors/error-response.ts`
  - mapped `Prisma.PrismaClientInitializationError` to:
    - status `503`
    - code `DATABASE_UNAVAILABLE`
    - message `The database is unavailable right now.`
- Test coverage:
  - added `tests/unit/core/errors/error-response.test.ts`
  - verified Prisma initialization failures are translated into the expected structured API response

## Validation

- Verified local Postgres was accepting connections on `localhost:5432`
- Verified app-style database connection:
  - `postgresql://postgres:postgres@localhost:5432/parallax_story_composer`
- Verified Prisma migration state is current
- Verified `POST /api/v1/guest-sessions` now returns a meaningful database-unavailable response when DB access is broken, instead of the generic unexpected-error response
- Verified focused unit test passes:
