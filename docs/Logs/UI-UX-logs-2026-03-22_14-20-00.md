# UI/UX Log Sheet

Timestamp: 2026-03-22 14:20:00

## Summary

Fixed a shared-header layout regression introduced after restoring `Settings` to primary navigation. The fourth nav item pushed the existing header layout past its tablet-width budget, which caused pages using the shared shell to render incorrectly.

## Error Observed

- Pages using the shared header no longer rendered correctly after `Settings` was added back to navigation
- The header layout became crowded at mid-width breakpoints
- Brand, nav, and actions competed for horizontal space before the mobile collapse breakpoint

## Root Cause

- The shared header was originally tuned for three primary nav links
- Adding `Settings` introduced a fourth nav item without updating the responsive layout rules
- `.psc-top-nav__links` stayed in a single row until `max-width: 720px`, which was too late for the expanded navigation set

## Fix Applied

- Updated `src/app/globals.css`
  - made `.psc-top-nav__inner` wrap cleanly at `max-width: 900px`
  - moved `.psc-top-nav__links` onto a full-width second row at tablet widths
  - reduced nav spacing and enabled horizontal overflow safety for the link row
  - added a header-only nav-hide breakpoint at `max-width: 820px`
  - preserved the original `max-width: 720px` page-shell breakpoint so landing, auth, projects, settings, and misc layouts keep their prior rendering behavior
- Kept the shared header wiring intact so `Settings` remains available in primary navigation

## Validation

- `npm run build` passed
- Verified the app routes still build successfully with the shared header in place
- Confirmed the responsive header rules now account for four navigation links

## Files Involved

- `src/app/globals.css`
- `src/features/shared/components/app-header.tsx`

## Notes

- This was a responsive layout regression, not a route or page-component failure
- An intermediate attempt moved a shared page-layout breakpoint too broadly; the final fix narrowed that change back to the header only to avoid regressions across the rest of the app shell
- The fix preserves `Settings` discoverability while restoring stable shared-shell rendering
