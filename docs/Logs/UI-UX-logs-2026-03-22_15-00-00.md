# UI/UX Log Sheet

Timestamp: 2026-03-22 15:00:00

## Summary

Applied a permanent shared-header fix by restoring a stable single-row header contract and updating the design docs to match the current app navigation and route structure.

## Error Observed

- Pages continued rendering incorrectly after adding `Settings` back into the shared header
- Earlier responsive fixes kept regressing because the shared header and page layouts were working from different assumptions
- Design docs were also still instructing a different navbar and route model than the app actually uses

## Root Cause

- Several layouts assumed a stable `72px` shared header height
- A previous responsive fix allowed the header to wrap, which broke those page-level height calculations
- Design docs still described `Product` and `Dashboard` labels and direct `/editor/[projectId]` and `/preview/[projectId]` routes, which conflicted with the current implementation

## Fix Applied

- Updated `src/app/globals.css`
  - introduced `--app-header-height` and `--editor-action-bar-height`
  - switched auth and editor viewport math to those shared variables instead of hardcoded values
  - restored the shared header to a stable single-row layout
  - kept the header responsive by tightening nav spacing and hiding nav links at a header-only breakpoint instead of changing broader page-shell breakpoints
- Left the current app navigation contract in place:
  - `Home` -> `/`
  - `Projects` -> `/projects`
  - `New Project` -> `/projects/new`
  - `Settings` -> `/settings`
- Updated design docs under `design/webpages/` to match the real app contract and route structure

## Validation

- `npm run build` passed
- Verified the app routes still build successfully with the permanent header contract in place
- Verified the design docs no longer require conflicting navbar labels or outdated editor/preview routes
