# UI/UX Log Sheet

Timestamp: 2026-03-22 14:00:00

## Summary

Restored `Settings` to the shared app header and aligned the settings page and MVP decision log with that navigation behavior.

## Error Observed

- The `/settings` route existed, but the shared header did not expose a `Settings` link
- The settings page rendered the shared header without an active Settings state
- Users could not discover Settings through primary navigation

## Root Cause

- `src/features/shared/components/app-header.tsx` only supported `home`, `projects`, and `new` active states
- The shared header rendered only `Home`, `Projects`, and `New Project`
- `src/features/settings/components/settings-page.tsx` rendered `<AppHeader />` without `active="settings"`
- `docs/setup/mvp-decision-log.md` still stated that Settings should stay out of primary navigation, which conflicted with the desired app-shell behavior

## Fix Applied

- Updated `src/features/shared/components/app-header.tsx`
  - added `settings` to the supported `active` union
  - added a `/settings` link to the primary navigation
  - wired active styling through `active === "settings"`
- Updated `src/features/settings/components/settings-page.tsx`
  - changed the shared header usage to `<AppHeader active="settings" />`
- Updated `docs/setup/mvp-decision-log.md`
  - clarified that `/settings` remains deferred in scope/content, but is still included in primary navigation as a discoverable route

## Validation

- `npx eslint src/features/shared/components/app-header.tsx src/features/settings/components/settings-page.tsx` passed
- `npm run build` passed
- Verified the shared navigation now includes `Settings`
- Verified the settings page can render with the Settings nav item marked active

## Files Involved

- `src/features/shared/components/app-header.tsx`
- `src/features/settings/components/settings-page.tsx`
- `docs/setup/mvp-decision-log.md`

## Notes

- This was a header wiring issue, not a missing route issue
- The settings page remains informational/deferred in MVP content, but it is now discoverable again through the shared app shell
